# 火山引擎豆包流式语音识别接入方案

## 定位

把教师的课堂录音、备课口述、教研讨论音频转成结构化文字，作为“一备多用适配器”的核心内容输入来源。

当前仓库的主 MVP 仍是 `Markdown + Prompt` 跑通“一份核心教学内容 -> 多个场景版本”。火山引擎豆包 ASR 是 `v1` 输入增强方案，不阻塞当前 Markdown MVP。

## 为什么选择字节流式 ASR

火山引擎豆包“大模型流式语音识别 API”更贴近这个项目后续的课堂场景：

- WebSocket 实时返回中间结果，适合边讲边转写。
- 资源 ID 明确；当前 MVP 实测使用 `volc.seedasr.sauc.duration`。
- Header 配置集中在 `API Key` 或 `App Key + Access Key`、`Resource Id`。
- MVP 已用普通双向流式 `bigmodel` 跑通；`bigmodel_async` 可作为后续优化项。
- 可以先用本地 WAV 文件模拟麦克风流，再接真实录音设备。

## 接口选择

| 场景 | 推荐接口 | 阶段 | 理由 |
|---|---|---|---|
| 边上课边生成文字 | 大模型流式语音识别 WebSocket | v1 首选 | 适合实时字幕和课堂记录 |
| 本地录音文件模拟实时输入 | 大模型流式语音识别 WebSocket | v1 验收 | 用 WAV 分片发送，验证识别质量 |
| 已有长录音批量整理 | 录音文件识别标准版 HTTP | v2 | submit/query 两阶段，适合长文件 |

## v1 流程

```text
教师录音
  -> 麦克风或本地音频转为 16kHz / mono / 16-bit WAV
  -> 建立 WebSocket: wss://openspeech.bytedance.com/api/v3/sauc/bigmodel
  -> 发送带 seq 的完整客户端请求
  -> 按 100-200ms 分片发送 gzip 压缩音频包
  -> 实时接收 partial / final 识别结果
  -> 汇总 result.text 和时间戳
  -> 清洗为 Markdown 核心内容输入
  -> 调用一备多用 Prompt 输出多个版本
```

## 必要配置

火山引擎侧需要准备：

- 开通豆包语音服务。
- 获取新版 `API Key`，或旧版 `APP Key` 和 `Access Key`。
- 确认大模型流式语音识别资源已开通。
- 使用资源 ID `volc.seedasr.sauc.duration`。

本地或服务端环境变量建议：

```bash
export VOLCENGINE_ASR_API_KEY="..."
export VOLCENGINE_ASR_WS_URL="wss://openspeech.bytedance.com/api/v3/sauc/bigmodel"
export VOLCENGINE_ASR_RESOURCE_ID="volc.seedasr.sauc.duration"
export VOLCENGINE_ASR_AUDIO_FORMAT="wav"
export VOLCENGINE_ASR_SAMPLE_RATE="16000"
export VOLCENGINE_ASR_CHUNK_MS="200"
export VOLCENGINE_ASR_UID="liumoxuan-teaching-magic"
```

如果你的控制台项目只提供旧版密钥，则改用：

```bash
export VOLCENGINE_ASR_APP_KEY="..."
export VOLCENGINE_ASR_ACCESS_KEY="..."
```

API Key / Access Key 只能放在服务端或本机私密环境变量中，不写入 README、样例文件、前端代码或 Git。

## WebSocket Header

| 字段 | 示例 | 说明 |
|---|---|---|
| `X-Api-Key` | `${VOLCENGINE_ASR_API_KEY}` | 新版控制台 API Key |
| `X-Api-App-Key` | `${VOLCENGINE_ASR_APP_KEY}` | 控制台 App Key |
| `X-Api-Access-Key` | `${VOLCENGINE_ASR_ACCESS_KEY}` | 控制台 Access Key |
| `X-Api-Resource-Id` | `volc.seedasr.sauc.duration` | 大模型流式 ASR 资源 ID |
| `X-Api-Connect-Id` | UUID | 每次连接唯一，便于排查 |
| `X-Api-Request-Id` | UUID | 每次请求唯一 |
| `X-Api-Sequence` | `-1` | 固定值 |

`X-Api-Key` 与 `X-Api-App-Key + X-Api-Access-Key` 按控制台密钥类型二选一。

## 初始化请求参数

连接建立后先发送完整客户端请求，payload 可按项目需要保持最小配置：

```json
{
  "user": {
    "uid": "liumoxuan-teaching-magic"
  },
  "audio": {
    "format": "wav",
    "rate": 16000,
    "bits": 16,
    "channel": 1,
    "codec": "raw"
  },
  "request": {
    "model_name": "bigmodel",
    "enable_itn": true,
    "enable_punc": true,
    "show_utterances": true
  }
}
```

初始化请求和音频包都需要按官方二进制协议封包，并对 payload 做 gzip 压缩。项目第一版可以直接复用官方 Demo 的协议封装，避免手写二进制细节。

## 最小响应字段

| 字段 | 用途 |
|---|---|
| `result.text` | 当前识别文本，作为 Markdown 初稿 |
| `is_last_packet` | 判断本轮音频是否结束 |
| `result.utterances[].text` | 分句文本，需要开启 `show_utterances` |
| `result.utterances[].start_time` | 分句开始时间 |
| `result.utterances[].end_time` | 分句结束时间 |
| `result.utterances[].definite` | 判断分句是否稳定 |
| `result.utterances[].words[]` | 词级结果，用于专名和错字修正 |
| 连接 ID / 日志 ID | 排查问题时提供给官方支持 |

## JSON 到核心内容 Markdown 的映射

| 转写结果 | 核心内容模板字段 | 处理方式 |
|---|---|---|
| 实时 `result.text` 高频主题词 | 教学主题 | 自动候选，人工确认 |
| 连续 partial/final 结果 | 讲解主线 | 合并碎句，删除口头禅 |
| “这道题”“这个例子”“看材料”附近文本 | 例题或文本材料 | 保留原始表述并标注 `[待核对]` |
| “容易错”“注意”“关键是”附近文本 | 易错点 | 优先保留，作为后续版本差异的核心依据 |
| “回去练”“作业”“请家长”附近文本 | 课堂任务 | 提取为任务清单 |
| 不顺或疑似错识别片段 | 待核对内容 | 保留时间戳，方便回听修正 |

## 接入前检查清单

- 输入音频优先使用 `16kHz / mono / 16-bit WAV`。
- MVP 脚本会用 `ffmpeg` 统一转码，再按固定时长分片发送。
- 音频包建议按 `100-200ms` 分片发送。
- 每次连接生成唯一 `X-Api-Connect-Id`。
- WebSocket 连接、发送音频、接收结果要拆成独立协程或线程。
- 需要处理中间结果覆盖、最终结果落盘、断线重连。
- 原始音频尽量不包含学生姓名、手机号、家庭信息、敏感评价。

## 热词策略

第一轮建议先维护一份学科热词表，实测发现错词后再决定是否接入官方热词能力或做后置修正。

```text
赤壁赋
一词多义
意象
函数单调性
韦达定理
```

## 数据安全策略

- API Key 只放服务端或本机 `.env`。
- `.env` 不提交 Git；`.env.example` 只保留空值。
- 音频上传前尽量脱敏，避免包含学生完整姓名和敏感评价。
- 使用临时 URL 时设置较短有效期。
- 转写 JSON 只保留必要字段；如需长期留存，放在本地受控目录或加密存储。
- 日志里不打印 API Key、签名 URL、完整学生信息。

## 最小验收样例

1. 录制一段 2-5 分钟真实教学口述。
2. 准备 `WAV` 或 `MP3` 文件。
3. 转成 `16kHz / mono / 16-bit WAV`。
4. 用流式 WebSocket 按 100-200ms 分片发送。
5. 汇总最终识别文本。
6. 人工检查以下指标：
   - `CER` 字错率。
   - 专名识别率。
   - 标点和分段可读性。
   - 例题条件是否完整保留。
   - 易错点和课堂任务是否能被提取。
   - 人工修正耗时是否低于从头整理。
   - 是否能稳定填入 `examples/input/示例-单份核心内容.md` 的结构。
7. 用转写文本生成至少 3 个输出版本。

## 推荐迭代顺序

1. `v0`：继续用 Markdown + Prompt 跑通核心产品闭环。
2. `v1`：配置火山引擎流式 ASR，跑通 `bigmodel` WebSocket 协议和最小音频分片。
3. `v2`：补脚本完成“本地音频文件 -> WAV 分片 -> 流式转写 Markdown”。
4. `v3`：把转写结果自动套入核心内容模板。
5. `v4`：接入真实麦克风，支持课堂实时记录。
6. `v5`：加入热词表、分段、说话人、摘要和任务提取。
