## 一、讨论结论

本项目从最初的“一份核心内容生成多个版本”，升级为“知识点库 + 学情库 + 资源库 + 当堂记录”的组合式备课系统。

核心目标仍然是“备一次，用在十个地方”，但“备一次”不再只是一段教案文本，而是一个可持续沉淀的知识点资产；“十个地方”也不只是多个文本版本，而是个性化导入、分层教案、练习生成、家长说明、课后反馈、复用案例等多个教学出口。

## 二、产品主线

1. 教师先建立或选择一个知识点。
2. 系统读取与知识点相关的教材、题库、教师讲法、注意点、附件资料。
3. 教师选择对象：单个学生、一个班级、或某一类学生。
4. 系统读取对象的学情：历史课堂记录、上节课反馈、学习地图、手动画像，或本次粘贴的转写文本。
5. 大模型根据“知识点资产 + 学情输入 + 资源库”生成个性化教学输出。
6. 输出可以组装为课前教案、课堂讲法、分层练习、自学材料、家长说明、课后反馈。
7. 用过的优质结果可以沉淀为复用案例，未来给新老师或同类学生参考。

## 三、关键共识

### 1. 不要只做静态改写器

仅把同一段内容改成三个语气版本，会显得不够贴近真实教学。更有价值的是：同一个知识点进入不同学生或班级的学情后，生成不同的讲法、导入、练习和反馈。

### 2. 知识点要成为可管理资产

知识点不是一次性输入。它应进入一个知识管理系统，教师可以在知识点下持续积累：

- 核心定义
- 讲法
- 常见误区
- 注意点
- 题目来源
- 教材页或 PDF
- 自己的补充说明
- 历史优秀案例

### 3. 学情可以来自多人、多班、多来源

学情可以按单个学生管理，也可以按班级或学生层次管理。系统要允许：

- 手动输入学生画像
- 从历史学生库读取学习地图
- 粘贴上节课录音转写
- 粘贴本次课堂录音转写
- 为没有历史记录的新学生临时建立画像

### 4. 录音解析先留接口，MVP 可先吃文本

讨论中提到可以用 STT 模型或外部 API 处理录音，但第一阶段没有必要把录音解析做成主任务。MVP 先支持“粘贴转写文本”，同时在实体模型中保留音频文件、转写文本、STT 任务的接口位。

### 5. 输出要能被前端组装和下载

大模型最好返回结构化 JSON，再由网页端解析并组装成页面。后续可以支持 Markdown、PDF、可打印讲义、家长反馈等下载格式。

### 6. 新东方或团队协作也可复用这套模式

这个项目形成后，可以迁移到新东方研发或其他多人协作场景：核心资料在仓库里集中维护，协作者拉取最新内容、在自己的分支上更新，再通过 PR 合并。

## 四、白板图解读

白板上可以整理为三层输入、一个生成中枢、多个输出。

### 三层输入

1. 主线输入：老师准备的主题或知识点。
2. 学情输入：学生 1、学生 2、班级、历史记录、学习地图。
3. 资源输入：题库、教材库、知识点库、历史案例。

### 生成中枢

- 上方标注为 `1+2 -> LLM -> json -> python -> app`。
- 可理解为：知识点与学情合并后进入大模型，大模型返回 JSON，后端或前端再组装为可视化页面和下载文件。

### 输出

- 个性化教案：包含约 10-15 分钟个性化导入、固定知识点讲解、个性化练习。
- 复用案例库：将可复用的教案、练习、讲法保存为案例。
- 反馈：本次录音或转写加知识点，生成课后反馈。
- 家长说明：用家长听得懂的话再说一遍。

## 五、待定问题

1. 第一版演示用哪个语文知识点：作文一材多用、文言实词、阅读概括，还是诗歌意象？
2. MVP 是否只做单学生，还是同时支持班级？
3. 第一版是否接真实 LLM API，还是先用手动 Prompt 和静态样例演示？
4. PDF 下载是否进入第一阶段，还是先只输出 Markdown/HTML？
5. 学生真实数据是否脱敏后才进入样例库？

## 六、下一步建议

1. 先做“知识点管理 + 学情粘贴 + 个性化教案输出”的闭环。
2. 第一版暂不做音频上传解析，只做“转写文本输入框”和 STT 预留位。
3. 选择一个真实语文知识点，准备 1 个知识点卡、2 个学生画像、1 段课堂转写。
4. 用结构化 JSON 固定输出格式，便于后续前端组装。
5. 把优秀输出沉淀到复用案例库，证明“备一次”会越来越值钱。
