## 一、核心问题

教师备课的高消耗，不只在于“想内容”，更在于同一份内容要被反复改写给不同对象：不同层次班级、请假学生、自学学生、家长、复习阶段、课后反馈。这个项目要解决的问题是：如何让一份真实核心教学内容，在不丢失教学主线的前提下，稳定生成多个可直接使用的版本。

---

## 二、设计原则

1. **一份核心，多个场景** — 项目不追求凭空生成内容，而是把教师已经确认过的核心内容扩展为多种版本。
   > 核心输入必须足够真实：可以是一份教案、一段讲义、一个知识点、一节课的讲授主线。

2. **先保真，再变体** — 每个输出版本都必须守住核心教学目标、关键概念、例题逻辑和易错点，再根据对象调整表达。
   > 版本之间可以换语气、换结构、换任务，但不能换掉知识中心。

3. **对象决定语言** — 不同版本的差异不只是长短，而是读者身份、知识基础、使用场景和行动目标的差异。
   > 给学生自学版要可操作，给家长版要可理解，给高阶班版要有挑战。

4. **矩阵化管理输出** — 用固定矩阵管理版本，避免每次临时想“还能改成什么”。
   > 第一阶段至少覆盖课堂分层版、自学版、家长说明版三类。

5. **可验证的减负** — 项目成败不看生成了多少文本，而看教师是否真的减少重复改写，并且生成物可直接进入课堂或沟通场景。
   > 每次样例都记录输入成本、输出版本、人工修改点和最终可用度。

## 三、关键决策记录

| 决策 | 选择 | 原因 | 放弃的方案 |
|------|------|------|-----------|
| 项目定位 | 做“一备多用适配器” | 截图任务明确要求“一个输入 -> 多个版本输出”，适合先做适配器而非完整备课系统 | 做完整备课平台；范围太大，容易偏离减负增效的核心证明 |
| 输入方式 | 先用结构化 Markdown 输入 | 方便教师粘贴已有教案、讲义、知识点，也方便后续转成表单或 Web 工具 | 一开始就做复杂表单；开发成本高，且会限制教师真实材料 |
| 输出策略 | 用版本矩阵控制输出 | 能明确对象、场景、难度、语气、长度和任务形式，减少随机改写 | 单条 Prompt 直接生成多个版本；质量不稳定，难以追踪差异 |
| 验证方式 | 用真实语文教学内容验证 | 项目服务于真实备课，必须用真实内容证明“能用在这么多地方” | 用虚构样例演示；展示好看但不能证明课堂可用 |

## 四、与传统方案的对比

| 维度 | 本项目 | 传统做法 | 优势 |
|------|--------|---------|------|
| 备课复用 | 一份核心内容生成多版本 | 每个场景手动重写一遍 | 降低重复劳动，保留同一条教学主线 |
| 版本管理 | 用矩阵记录对象、难度、场景 | 靠教师临场判断 | 版本差异更清楚，后续可复盘 |
| 输出质量 | 先校验核心，再生成变体 | 写完一个版本再凭经验改 | 更容易避免核心偏移 |
| 迭代方式 | 记录修改点，反向优化 Prompt | 每次从头调整措辞 | 能积累为稳定工具 |

## 五、适用边界

### 适用场景

- 同一节课要给不同层次班级使用。
- 同一知识点要同时生成课堂讲法、自学材料、家长说明。
- 学生请假后，需要一份可独立补学的版本。
- 复习课、反馈课、讲义改写中需要保留核心但更换表达。

### 不适用场景

- 原始核心内容本身尚未成立，需要先做教研设计。
- 需要生成全新课程体系，而不是复用已有内容。
- 涉及精密考试答案或官方口径时，尚未进行人工核对。
- 输出对象和使用场景完全不清楚，只要求“多写几个版本”。
